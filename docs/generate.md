# mcp-workbench generate

## Overview

The `generate` command connects to a live MCP server, discovers its capabilities (tools, resources, prompts), and outputs a YAML test spec scaffold you can use with `mcp-workbench run`.

**Why it exists:** Writing test specs from scratch requires knowing every tool name, argument shape, and URI ahead of time. `generate` eliminates that bootstrap cost by introspecting the server directly.

**What it produces:** A starting point, not a finished test suite. Every generated spec needs human review — argument placeholders (`"TODO"`) are intentional signals that the spec is incomplete.

### Recommended workflow

```
generate → review/edit → run → snapshot/diff → refine
```

1. `generate` — discover the server, produce a skeleton spec
2. Review the YAML, replace `"TODO"` values with real inputs
3. `run` — execute the spec against the server
4. Examine snapshot diffs and assertion results
5. Tighten assertions, add edge cases, repeat

---

## Usage

### Basic examples

```bash
# HTTP server
mcp-workbench generate --transport streamable-http --url http://localhost:8081/mcp -o tests.yaml

# stdio server
mcp-workbench generate --transport stdio --command node --args "dist/server.js" -o tests.yaml

# With authentication
mcp-workbench generate --transport streamable-http --url http://localhost:8081/mcp \
  --header "Authorization: Bearer sk-xxx" -o tests.yaml

# stdout only (pipe or preview)
mcp-workbench generate --transport streamable-http --url http://localhost:8081/mcp --stdout

# Filter specific capabilities
mcp-workbench generate --transport streamable-http --url http://localhost:8081/mcp \
  --include tools,resources -o tests.yaml
```

## Options

### Connection

| Option | Description |
|--------|-------------|
| `--transport <kind>` | `stdio` or `streamable-http` (required) |
| `--url <url>` | Server URL (HTTP transports) |
| `--command <cmd>` | Server executable (stdio) |
| `--args <args>` | Arguments passed to the server command (stdio) |
| `--header <key: value>` | HTTP header to include with every request (repeatable) |
| `--timeout <ms>` | Connection timeout in milliseconds (default: 10000) |

### Generation

| Option | Default | Description |
|--------|---------|-------------|
| `--include <list>` | all | Comma-separated subset to generate: `tools`, `resources`, `prompts` |
| `--exclude <list>` | none | Comma-separated capabilities to skip |

### Output

| Option | Description |
|--------|-------------|
| `-o, --output <file>` | Write the generated spec to a file |
| `--stdout` | Print the generated spec to stdout instead of a file |
| `--overwrite` | Overwrite an existing output file (default: error if file exists) |

At least one of `--output` or `--stdout` is required.

## What Gets Generated

### Always included

Every generated spec contains:

- `apiVersion: mcp-workbench.dev/v0alpha1` header
- A `server` block that mirrors the connection parameters you provided
- List tests for each capability the server advertises (tools/list, resources/list, prompts/list)

### Per-tool tests

For each tool discovered via `tools/list`:

- A `tools/call` test case named after the tool
- `status` assertion (`"success"`)
- `executionError` assertion (expects `null`)
- `args` block populated from the tool's `inputSchema`:
  - Required fields get type-inferred placeholder values
  - Fields that cannot be safely inferred get `"TODO"`

### Per-resource tests

For each resource returned by `resources/list`:

- A `resources/read` test case
- URI set to the discovered resource URI
- No assertions beyond HTTP-level success (add your own after review)

### Per-prompt tests

For each prompt returned by `prompts/list`:

- A `prompts/get` test case
- `args` block with `"TODO"` for each required argument

## Example Output

Given a server with two tools (`get_weather`, `search`), one resource (`file:///data/readme.txt`), and one prompt (`summarize`), `generate` produces:

```yaml
# Generated by: mcp-workbench generate
# Server: my-server v1.0.0 (protocol 2025-11-25)
# Review and customize before using in CI.

apiVersion: mcp-workbench.dev/v0alpha1

server:
  transport: streamable-http
  url: http://localhost:8081/mcp

tests:
  - id: tools-list
    description: Server exposes tools
    act:
      method: tools/list
    assert:
      - kind: status
        equals: success
      - kind: notEmpty
        path: $.tools

  - id: tool-get-weather
    description: "Tool: get_weather"
    act:
      method: tools/call
      tool: get_weather
      args:
        city: "Seoul"
    assert:
      - kind: status
        equals: success
      - kind: executionError
        equals: false

  - id: tool-search
    description: "Tool: search"
    act:
      method: tools/call
      tool: search
      args:
        query: "test"
        limit: 10
    assert:
      - kind: status
        equals: success
      - kind: executionError
        equals: false

  - id: resources-list
    description: Server exposes resources
    act:
      method: resources/list
    assert:
      - kind: status
        equals: success
      - kind: notEmpty
        path: $.resources

  - id: resource-readme
    description: "Resource: file:///data/readme.txt"
    act:
      method: resources/read
      uri: "file:///data/readme.txt"
    assert:
      - kind: status
        equals: success

  - id: prompts-list
    description: Server exposes prompts
    act:
      method: prompts/list
    assert:
      - kind: status
        equals: success

  - id: prompt-summarize
    description: "Prompt: summarize"
    act:
      method: prompts/get
      name: summarize
      args:
        text: "TODO"
    assert:
      - kind: status
        equals: success
```

## Placeholder Policy

When an argument value cannot be inferred safely, `generate` writes the literal string `"TODO"`.

**Why `"TODO"` instead of an empty string or `null`?**

An empty string or null might accidentally be a valid input, allowing the test to pass silently with incorrect data. `"TODO"` is almost never a valid value, so the test fails loudly — forcing you to supply a real value before the spec is considered complete.

### Type inference rules

| Condition | Generated value |
|-----------|-----------------|
| `type: string`, field name contains `city` | `"Seoul"` |
| `type: string`, field name contains `query` or `search` | `"test"` |
| `type: string`, field name contains `name` | `"example"` |
| `type: string`, field name contains `url` | `"https://example.com"` |
| `type: string`, field name contains `path` or `file` | `"/tmp/example"` |
| `type: string` (generic) | `"example"` |
| `type: number` or `type: integer` | `1` |
| `type: boolean` | `true` |
| `enum` | First value in the enum array |
| Unknown / complex schema | `"TODO"` |

Inference is shallow. Nested object schemas and `oneOf`/`anyOf` always produce `"TODO"`.

## Limitations (v0.1)

- **No deep mode:** `generate` only calls `tools/list`, `resources/list`, and `prompts/list`. It does not execute any tools or read any resources to discover realistic outputs.
- **No AI-assisted arg generation:** Placeholder inference is purely heuristic, not LLM-powered.
- **No merge with existing specs:** Running `generate` twice does not update an existing file — use `--overwrite` to replace it entirely.
- **No destructive tool detection:** Tools with names suggesting side effects (`delete`, `drop`, `reset`, etc.) are included without warning. Detection and skip logic is planned for v0.2.
- **Shallow schema inference only:** Complex input schemas (nested objects, `oneOf`, `anyOf`, `$ref`) always fall back to `"TODO"`.

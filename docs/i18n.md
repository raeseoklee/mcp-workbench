# Internationalization (i18n)

MCP Workbench supports multiple output languages for its CLI.

---

## Design philosophy

- **Zero dependencies** — locale data is inlined at build time; no runtime file I/O.
- **Browser-compatible** — the `@mcp-workbench/i18n` package works in Node.js and in browser-based UIs.
- **Flat key namespace** — translation keys are dot-separated strings (`cli.run.header`).
- **English fallback** — missing keys in any locale transparently fall back to English, then to the key itself.
- **Only user-facing strings** — protocol payloads, JSON output, assertion keys, and YAML spec keys are always in English.

---

## Supported languages

| Locale code | Language |
|-------------|----------|
| `en`        | English (default) |
| `ko`        | Korean |

---

## Language selection

Priority order (highest → lowest):

1. `--lang <locale>` CLI flag
2. `MCP_WORKBENCH_LANG` environment variable
3. Default: `en`

```bash
# Flag (highest priority)
mcp-workbench run tests.yaml --lang ko

# Environment variable
MCP_WORKBENCH_LANG=ko mcp-workbench inspect --command mcp-workbench-demo

# Pipe-friendly: env var + JSON output (output unchanged)
MCP_WORKBENCH_LANG=ko mcp-workbench run tests.yaml --json | jq .
```

## Translation key conventions

Keys use dot-separated namespaces:

| Prefix        | Scope |
|---------------|-------|
| `cli.run.*`     | `mcp-workbench run` output (headers, labels, summaries) |
| `cli.inspect.*` | `mcp-workbench inspect` output (server info, capabilities) |
| `errors.*`      | (reserved for future shared error messages) |
| `status.*`      | (reserved for future status strings) |

### Naming rules

- Use lowercase, dot-separated segments.
- Use nouns for labels (`cli.run.label.spec`), verbs for actions (`cli.run.error.loadSpec`).
- Keep keys descriptive enough to find without reading the value.

### Param placeholders

Use `{paramName}` for interpolated values:

```json
"cli.run.error.loadSpec": "Failed to load spec: {file}"
```

```typescript
t("cli.run.error.loadSpec", { file: "tests.yaml" })
// → "Failed to load spec: tests.yaml"
```

## How to add a new locale

1. **Copy the canonical JSON**

   ```bash
   cp packages/i18n/locales/en.json packages/i18n/locales/<locale>.json
   ```

2. **Translate all values** — keep all keys identical; only change the values.

3. **Add the locale to `packages/i18n/src/i18n.ts`**

   ```typescript
   import fr from "./locales/fr.js"; // if using TS modules

   // Or add the object inline, mirroring the JSON:
   const fr: LocaleDict = {
     "cli.run.header": "MCP Workbench",
     "cli.run.error.loadSpec": "Impossible de charger le spec : {file}",
     // … all keys
   };
   ```

   Register it in the `LOCALES` map:

   ```typescript
   const LOCALES: Record<string, LocaleDict> = { en, ko, fr };
   ```

4. **Build and test**

   ```bash
   pnpm --filter @mcp-workbench/i18n build
   pnpm --filter @mcp-workbench/i18n test
   ```

5. **Update this document** — add the new locale to the table above.

## Package structure

```
packages/i18n/
  locales/
    en.json          ← canonical English source (kept in sync with src/i18n.ts)
    ko.json          ← canonical Korean source
  src/
    i18n.ts          ← inline locale data + t(), setLang(), getLang()
    index.ts         ← public re-exports
    __tests__/
      i18n.test.ts   ← language switching, fallback, interpolation tests
```

> **Note:** Locale data is intentionally inlined in `src/i18n.ts` rather than imported from the JSON files.
> This keeps the package browser-compatible without requiring a bundler or Node.js file system access.
> The JSON files in `locales/` are the canonical human-editable source; update both when making changes.

## Using i18n in code

```typescript
import { t, setLang, getLang, getSupportedLocales } from "@mcp-workbench/i18n";

// Set language (e.g. from --lang CLI flag)
setLang("ko");

// Translate a key
console.log(t("cli.run.header"));
// → "MCP Workbench"

// Translate with params
console.log(t("cli.run.error.loadSpec", { file: "tests.yaml" }));
// → "스펙 파일 로드 실패: tests.yaml"

// Inspect current locale
getLang();              // → "ko"
getSupportedLocales();  // → ["en", "ko"]
```

The `MCP_WORKBENCH_LANG` environment variable is read at module load time and sets the initial language automatically.
